# DECISIONS.md — 設計判断の記録（ADR）

このファイルは「**なぜその選択をしたか / 何を比較して何を却下したか**」を記録する。SPEC.md が「今の仕様（What）」、本ファイルが「その理由と却下案（Why）」を担う。仕様の蒸し返しや、確定事項の意図しない作り替えを防ぐのが目的。

各エントリの形式: **決定 / 理由 / 却下した代替案**。

---

### D-001. X（Twitter）を収集対象から除外
- **決定**: MVP・当面ともに X は収集しない。
- **理由**: X の公式 API は 2023 年以降有料かつ高額で、無料枠は実用にならない。スクレイピングは規約違反かつ不安定。費用対効果が見合わない。
- **却下案**: 有料 API 契約（コスト過大）、Nitter 等の非公式手段（規約・安定性のリスク）。

### D-002. X の代替として Qiita / Zenn / 公式 RSS を採用
- **決定**: X を外す代わりに、Qiita API・Zenn RSS・公式ブログ/GitHub の RSS/Atom を主軸にする。
- **理由**: いずれも取得が容易で安定。とくに GitHub Releases の `.atom` と Qiita API は堅牢で、一次情報・日本語コミュニティ情報の両方を網羅できる。むしろ X 依存より網羅性が上がる。

### D-003. YouTube は将来実装（MVP 非対象）
- **決定**: MVP には含めず、後から追加する。
- **理由**: 新着動画取得は可能だが、内容の要約に字幕取得/文字起こしが別途必要で MVP のスコープを膨らませる。ソースアダプタ設計により後付けが容易なため、急がない。

### D-004. `anthropic.com/news` を「もろい前提」で採用
- **決定**: 第三者スクレイピング由来のフィードで暫定的に取得し、MVP に含める。
- **理由**: コンテンツ自体は Anthropic 公式・正確。Anthropic は公式 RSS を提供していないため、配信の器は第三者製にならざるを得ない。情報価値が高いので、脆さを承知で採用する。
- **却下案**: 採用見送り（公式一次情報を逃す）。即時の自前スクレイピング/RSSHub 自前ホスト（MVP には重い）。
- **補足**: 運営停止・構造変更で停止し得る。アダプタ抽象化により、後から RSSHub 自前ホストや自前スクレイピングへ差し替え可能にしておく。

### D-005. 記事の分類は「タイトル単語」ではなく「内容」で判定
- **決定**: update / tips の振り分けは LLM が内容を見て判定する。
- **理由**: タイトルのキーワード一致だけでは誤判定が多い。例：「最新」を含む活用術記事を update と誤る、新モデル名を含む検証記事を update と誤る。「`gemini-3-flash` を試した」は新モデル名を含んでも、プロダクトの告知ではないため `tips` が正。
- **却下案**: タイトルのキーワードマッチのみ（0 コールで無料だが精度が粗い）。どのみち要約で全記事を LLM に通すため、同じ 1 コールに分類を相乗りさせるのが最適。

### D-006. 1 記事の加工は LLM 1 コールに集約
- **決定**: 要約・分類・重要度スコアを 1 回のプロンプトで JSON 取得する。
- **理由**: 3 つを別々に呼ぶとコール数が 3 倍になり、無料枠と速度を圧迫する。要約のために本文を渡す“ついで”に分類とスコアも出させればコストはほぼ増えない。

### D-007. 重複排除キーは「URL 正規化」を採用（external_id ではなく）
- **決定**: 一意キーは正規化後 URL。`source` と `external_id` はカラムとして保持するが一意キーには使わない。
- **理由**: ソースが振る ID（Qiita の item id、Zenn の slug 等）は体系がソースごとに別物で、横断では衝突・無意味になりうる。URL はグローバルに一意で、同一記事が複数ソース（例: はてブと Qiita）から流れても最終リンク先が同じなら 1 件にまとまる。
- **却下案**: `(source, external_id)` の複合キー（ソース間の実体重複を防げない）。生 URL そのまま（utm 等のパラメータ差で別記事扱いになる → 正規化で解決）。

### D-008. LLM は Gemini 無料枠を採用し、プロバイダ抽象化で差し替え可能に
- **決定**: MVP は Gemini 無料枠。`LLM_PROVIDER` の切替で Claude API に移行できる設計。
- **理由**: 無料で要約品質も実用的。将来品質を上げたくなった時に、入出力 JSON を固定したプロバイダ抽象化があれば、呼び出し側を変えずに移行できる。
- **却下案**: 最初から Claude API（無料枠がなく従量課金。MVP には過剰）。

### D-009. Gemini は Flash 主・Flash-Lite フォールバック
- **決定**: 主に Flash（250 RPD）を使い、枠を使い切ったら Flash-Lite（1,000 RPD）へ自動切替（合計約 1,250 RPD）。Flash は約 6 秒間隔で RPM 制御。
- **理由**: 品質の高い Flash を優先しつつ、無料枠がモデル別であることを活かして上限を実質拡張。日次 20〜50 件は Flash だけで収まるが、初回バックフィルや話題集中日の保険になる。
- **却下案**: Flash-Lite 単独（品質が落ちる）。Flash 単独（250/日 上限で初回や急増時に詰まる）。
- （追記）**D-024 により無効化**。2025-12 の無料枠削減で前提が崩れたため、Gemini 3.1 Flash-Lite 単独（15 RPM / 500 RPD）へ変更。

### D-010. LLM へ渡す本文は 2,000〜3,000 文字で打ち切る
- **決定**: 本文は先頭から上限文字数で truncate して渡す。
- **理由**: 記事は前半に要点が集まるため要約には十分。プロンプト肥大化・トークン制限超過・コスト/速度悪化を防ぐ。
- **却下案**: 本文全文を渡す（プロンプトが過大化し、トークン側の制限に当たりやすい）。タイトルのみ（内容判定の精度が落ちる、D-005 と矛盾）。

### D-011. 「今日の重要 TOP5」は LLM 重要度スコアを主、指標は補助
- **決定**: importance_score 降順で上位 5 件。同点時のみ Qiita ストック数・GitHub の新しさ・published_at でタイブレーク。
- **理由**: 公式ブログや GitHub Releases には人気指標が存在せず、指標ベース単独では一次情報の速報が埋もれる。LLM 判定なら指標のないソースも横断評価できる。
- **却下案**: 指標ベース単独（指標のないソースを評価できず、新着一次情報を取りこぼす）。

### D-012. 保持は「非お気に入り 30 日で物理削除、お気に入り永久」
- **決定**: `is_favorite = false` かつ published_at が 30 日より古い記事を物理削除。お気に入りは削除対象外。
- **理由**: 判断軸は容量ではなく鮮度。Supabase 無料枠の 500MB に対し本アプリのデータは軽く、容量は制約にならない。「1 日 1 回キャッチアップ」目的では数週間より前は見返さないため 30 日で十分。論理削除は自分専用では不要で、物理削除で DB を軽く保つ。
- **却下案**: 全件永久保持（鮮度低下・無駄な蓄積）。論理削除（自分専用ではメリットが薄い）。
- （追記）**D-028 / D-029 により更新**。お気に入りは localStorage スナップショットへ移行し、cleanup は `is_favorite` を参照せず「published_at 30 日超は全削除」に単純化。「永久保有」の意図はクライアント側で維持される。

### D-013. ソース登録は `.ts` ファイルに集約
- **決定**: `batch/src/sources/index.ts` に各アダプタと設定を配列で集約。追加削除は配列の 1 行増減で完結。
- **理由**: 登録リストはアダプタ（関数）への参照を持つためコードである必要があり、`product` 等の値の型チェックも効かせたい。
- **却下案**: JSON 設定（関数を参照できず、型チェックも効かない）。

### D-014. 公開日の表示は日付のみ（時刻なし）
- **決定**: 記事カードの日付は `published_at` の日付部分のみを表示（例「2026/06/14」）。
- **理由**: 「いつの情報か」が一目で分かれば十分で、時刻の精度は不要。表示をシンプルに保つ。

### D-015. バッチ基盤は GitHub Actions、ログは実行ログ（+ 任意で Supabase）
- **決定**: 収集・クリーンアップは GitHub Actions の cron。ログは `console` 出力（= GitHub Actions 実行ログ）を基本とする。
- **理由**: 実装経験があり手に馴染む。バッチ実行環境は使い捨てのためファイルログ（error.log 等）は永続しない。実行ログに出すのが構成に合う。後から実行サマリを Supabase のログテーブルに残す拡張も可能。
- **却下案**: ファイルベースのログ（使い捨て環境で永続しない）。常時稼働サーバ（オーバースペック）。

### D-016. フロントとバッチはモノレポ（軽量構成）
- **決定**: 1 リポジトリに `web/` と `batch/`（必要なら `shared/`）を同居。Nx/Turborepo 等の専用ツールは使わない軽量構成。
- **理由**: 記事の型などをフロント・バッチで共有でき、front/batch の変更を 1 コミットで整合させられる。規模的に専用ツールは過剰。
- **却下案**: リポジトリ分割（型の二重管理・整合ズレのリスク）。

### D-017. お気に入りは独立タブで表示
- **決定**: お気に入り（`is_favorite = true`）は独立したタブとして用意する。
- **理由**: 永久保有したストックを一覧で見返す動線を明確にするため。モーダルや導線リンクより、他のタブと並ぶ独立タブの方が常設の入口として分かりやすい。
- **却下案**: モーダル/サブ導線（常設性が弱く見返しにくい）。
- （追記）独立画面の方針は維持。データソースのみ `is_favorite`（DB）→ localStorage スナップショットへ変更（D-028）。

### D-018. 重要度バッジを UI に表示
- **決定**: 記事カードに重要度（importance_score）をバッジとして表示する。
- **理由**: TOP5 以外のタブでも、ひと目で重要度の高い記事を見分けられるようにするため。スコアは LLM が全記事に付与済みで追加コストはない。

### D-019. TOP5 の対象期間は「当日のみ」
- **決定**: 「今日の重要 TOP5」は公開日が本日の記事のみを対象にする。
- **理由**: 「1 日 1 回見れば追いつく」という設計思想に最も忠実。直近 N 日にすると過去の記事が居座り、当日の新着が埋もれて毎日の鮮度が落ちる。
- **却下案**: 直近 N 日（当日の新着が過去記事に押し出される）。
- （追記）D-023 により「直近24時間のローリングウィンドウ」へ更新。バッチが1日1回実行のため暦日境界だと新着が永久に表示されない欠陥が判明したため。
- （追記）D-036 により画面名を「重要トピック」へ改称し、選定を「importance 上位 5 件」から「ボタン5グループ別の当日最重要1件ずつ・全ユーザー一律」へ変更。24 時間ウィンドウは維持。

### D-020. 実物確認に基づくソース別の取り込み調整
- **決定**: 2026-06-14 の実物確認を踏まえ、(a) gemini-cli の releases.atom は nightly/preview を除外し安定版に絞る、(b) DeepMind Blog RSS と Google Developers Blog は Gemini/Gemma 関連のみキーワードフィルタで残す、(c) anthropic.com/news は Olshansk 配布の第三者フィードを採用、(d) Google Developers Blog の正確なフィード URL は実装時に最終確認する。
- **理由**: gemini-cli は nightly が高頻度で出るためノイズになる。DeepMind/Google ブログは AI 全般で対象外記事が多い。anthropic.com/news は live・本日更新を確認できた唯一の実用フィードだった。
- **却下案**: gemini-cli を全リリース取り込み（nightly でフィード汚染）。Google/DeepMind を無フィルタ取り込み（無関係記事の混入）。

### D-021. 画面構成を「タブ切替」から「ホーム画面 + ボタン遷移」に変更
- **決定**: 単一ページ内のタブ切替ではなく、ロボットマスコットと一言サマリーを表示するホーム画面を起点に、5 つの円形ボタン（TOP5 / アップデート / Tips:Claude Code / Tips:Gemini / お気に入り）から各画面へ遷移する構成にする。各画面には「← ホーム」の戻り導線を設ける。
- **理由**: UI デザイン検討の結果、ホーム画面でその日の状況を一言で把握できることが「1 日 1 回見るだけでキャッチアップが完了する」というコンセプトに合致すると判断した。タブ常設よりも、まずホームで全体感を掴んでから必要な画面に進む流れの方が「引き算」の体験に近い。
- **却下案**: 単一ページ + タブ切替（当初案。常時 5 つのタブが見えることで「全部見なければ」という圧を感じやすく、引き算思想との相性がタブより劣ると判断）。
- （追記）**D-027 により更新**。ホーム + ボタン遷移の構成は維持しつつ、ボタンは固定 5 個から「固定枠 2（TOP5 / お気に入り）+ 自由枠最大 4（ScreenConfig によるデータ駆動）」へ変更。

### D-022. テストフレームワークは Vitest を採用（フロント・バッチ共通）
- **決定**: フロント（`src/`）と `batch/` の両方で Vitest をテストフレームワークとして採用する。フロントは `jsdom` 環境、バッチは `node` 環境で動作する。
- **理由**: フロント・バッチともに Vite / TypeScript 構成のため Vitest はほぼゼロ設定で導入できる。具体的には (a) `vitest/config` の `mergeConfig` でフロントの Vite 設定を再利用でき、バンドラ設定とテスト設定を二重管理しない、(b) `"type": "module"` の ESM プロジェクトとネイティブに相性が良い、(c) `vi.mock` / `vi.spyOn` がビルトインで外部依存（Supabase・Gemini API・RSS fetch 等）のモックが容易、(d) watch・カバレッジが 1 パッケージで揃う。
- **却下案**: Jest（CommonJS 前提でありESM/Vite 構成との相性が悪く、`babel-jest` や `ts-jest` の変換設定が別途必要）。Bun Test（将来の選択肢だが Node.js 20 ベースのバッチとの整合性でリスクあり。このプロジェクトでは時期尚早）。

### D-023. TOP5の対象期間を「当日暦日」から「直近24時間」に変更
- **決定**: 「今日の重要TOP5」の対象は published_at が現在時刻から遡って24時間以内の記事とする（暦日境界ではなくローリングウィンドウ）。あわせて collect.yml のバッチ実行時刻を JST 7:00 → JST 5:00（`cron: '0 20 * * *'` UTC）に変更する。
- **理由**: バッチは1日1回のみ実行のため、実行時刻〜23:59に公開された記事は翌日のバッチで収集された時点で published_at が「前日」扱いとなり、暦日境界の当日フィルタ（D-019）には永久に一致しなかった。実行時刻をずらしても同じ欠陥が時間帯を変えて残るだけで解決しない。ローリング24時間なら実行タイミングに依存せず、収集さえされていれば直近の新着を取りこぼさない。バッチ実行時刻はJST 5:00に早め、当日分の収集をより早いタイミングで完了させる。
- **却下案**: バッチ頻度を上げる（1日数回実行）。効果はあるが、GitHub Actionsの実行回数を増やす運用コストが増え、根本原因（暦日境界という固定点）は残ったまま回避するだけ。今回はクエリ側の修正のみで解決できるため見送り。
- **補足**: D-019が却下した「直近N日ウィンドウ」（古い記事の居座りで新着が埋もれる懸念）とは実質的に衝突しない。ウィンドウ幅は24時間＝1日相当のままで、単に固定の暦日境界を撤廃しただけであり、鮮度を損なわない。

### D-024. LLM は Gemini 3.1 Flash-Lite 単独、設計値は枠の約 56%
- **決定**: Gemini 3.1 Flash-Lite（15 RPM / 500 RPD）を使用。日次収集上限は合計約 280 件とし、約 4 割の余白を残す。RPM 制御は 1 件ごとに約 4 秒間隔。
- **理由**: 2025-12 の無料枠削減により旧前提（Flash 250 + Flash-Lite 1,000。D-009）が無効化。余白は枠の再削減・リトライ・同日再実行（RPD リセットは JST 16 時）への保険。
- **却下案**: 枠の 9 割まで使う設計（上記 3 リスクのどれかで即詰まる）。

### D-025. 分類軸を category 単軸から「category × audience × difficulty」の直交 3 軸に拡張
- **決定**: LLM 出力に `audience`（engineer/backoffice/executive）と `difficulty`（1〜3）を追加。同一 1 コールに相乗りさせる。`category` に `business` / `case_study` を追加。
- **理由**: 属性別の表示分けを追加コストゼロで実現するため。複合ラベルを単一 enum にすると属性追加時に組み合わせが爆発する。
- **却下案**: 複合 enum（組み合わせ爆発）。属性別に要約を作り分ける（1 記事 × 属性数のコールで枠が破綻）。

### D-026. カード可読性向上は enrich 1 コールへのフィールド追加で実現
- **決定**: `importance_reason`（重要な理由の短文）と `tags`（2〜3 個）を LLM 出力に追加し、カードに表示する。
- **理由**: 「重要度 7」だけでは何にとっての重要度か判断できない。既存コールへの相乗りで追加コストは実質ゼロ。
- **却下案**: 既存記事のバックフィル（不要。30 日ローテーションで自然に入れ替わる）。

### D-027. カスタム画面は ScreenConfig によるデータ駆動 + 固定枠 2 + 自由枠最大 4
- **決定**: 画面 = クエリ定義（ScreenConfig）としてデータ化し、設定画面のチェックボックスでホームのボタンを選択可能にする。TOP5 とお気に入りは固定枠、自由選択は最大 4 つ。
- **理由**: チェックの粒度を product × category × audience × difficulty に揃えると、属性プリセットも新プロダクト追加も設定の組み合わせで完結する。上限 4 は選びすぎ（足し算への回帰）の防止。
- **却下案**: 画面のハードコード継続（カスタム不可）。無制限チェック（引き算思想に反する）。
- （追記）**D-035 により更新**。自由枠は最大 5（合計最大 7 ボタン）へ変更し、ボタンはグループ単位で統合生成する方式に。

### D-035. 設定画面はグループ単位のボタン統合方式、合計最大 7 ボタン（2026-07-04 デザイン確定）
- **決定**: UI デザイン（claude.ai/design「Button Settings.dc.html」）の確定に伴い、(a) ホームのボタンは固定 2（TOP5 / お気に入り）+ 自由枠最大 5 の**合計最大 7 個**、(b) ボタンはチェックボックス単位ではなく**グループ単位**（Gemini / Claude Code / Codex / バックオフィス / 経営者向け の 5 グループ）で生成し、同一グループ内の複数チェックは 1 ボタンに統合、(c) フィルタの `difficulty` / `category` は配列化して `.in()` でクエリ、(d) 独立した「アップデート」グループは廃止し、update 記事はエンジニア系ボタンに難易度別で吸収（リリースノート読み解きは §4.3 どおり上級に寄る）。
- **理由**: 難易度ごとに別ボタンだとボタンが乱立し「足し算」に回帰する。グループ統合なら「初級だけ」も「全難易度まとめて」も同じ 1 ボタンで表現でき、ボタン総数が構造的に 7 を超えない（上限ロジックがほぼ不要になる）。LLM 出力・DB スキーマは D-025 の 3 軸のまま変更不要で、フロントのフィルタ合成だけで実現できる。
- **却下案**: チェック単位でボタン生成（最大 13 ボタンに乱立しうる）。アップデート独立グループの維持（デザイン検討の結果、リリースノートは難易度軸に吸収する方が画面数を減らせる）。LLM に複合ラベルを出させる（D-025 で却下済みの方式に逆戻り）。

### D-028. ユーザー設定・属性・お気に入りは localStorage 管理
- **決定**: お試し段階の複数人利用では、設定/属性/お気に入りを localStorage に保存。お気に入りは★押下時に記事オブジェクト全体をスナップショット保存し、お気に入り画面は localStorage から描画する。`articles.is_favorite` は非推奨（カラムは残すが参照しない）。
- **理由**: 認証（メール & パスワード）の手間を避けたい段階のため。`articles.is_favorite` は全員共有のグローバル値になり複数人では成立しない。スナップショット方式なら DB の物理削除と独立して「永久保有」（D-012 の意図）を維持できる。
- **却下案**: is_favorite 継続（複数人で衝突）。Supabase 認証 + favorites テーブル（お試し段階では過剰。将来の本格対応時の移行先とする）。

### D-029. 物理削除は 30 日を維持（7 日案は却下）
- **決定**: 保持期間は 30 日のまま。cleanup の条件から is_favorite 参照を外し「30 日超は全削除」に単純化。
- **理由**: RSS/Atom フィードには数週間分の過去記事が残るため、7 日削除だと「削除 → 新規と誤判定 → 再収集・再 enrich」のゾンビループが発生する。
- **却下案**: 7 日削除 + fetch 時の日付フィルタ（30 日維持ならフィルタ自体が不要で、よりシンプル）。

### D-030. 表示は日数ウィンドウではなく件数上限（リスト 20 件）
- **決定**: 各リスト画面は published_at 降順の最新 20 件。ページネーションなし。~~TOP5 の選定対象はチェック中カテゴリの記事に絞る（フロント側フィルタ）。~~
- **理由**: 「当日+前日」の日数方式は過疎カテゴリ（Codex・導入事例）で空画面が構造的に発生する。20 件は活況カテゴリで約 2 日分に相当し当初意図を保ちつつ、過疎カテゴリでは自動的に過去へ遡って埋まる。
- **却下案**: 当日+前日表示（空画面が発生）。無限スクロール（引き算思想に反する）。
- （追記）**D-036 により TOP5 の絞り込み部分を撤回**。「重要トピック」はチェック中カテゴリではなくボタン5グループ別の当日最重要1件ずつ・全ユーザー一律に変更（リスト 20 件の方針は維持）。

### D-031. クエリ別日次上限 + ソース固有指標による優先拾い（rank）
- **決定**: enrich に回す件数をクエリ単位で制限（合計約 280/日、`SourceConfig.dailyLimit`）。上限超過時の選抜は、指標のあるソース（Qiita: ストック数、はてブ: ブクマ数）のみ人気降順、それ以外は新着順。カット件数を実行ログに出力する。初回バックフィル対策（旧「初回のみ期間を絞る」方式）もこの上限が兼ねる。
- **理由**: importance_score は enrich の出力のため選抜には使えない（順序が逆）。enrich 前に無料で得られる指標のみ活用。
- **却下案**: 全件 enrich してスコアで選抜（本末転倒）。全ソース新着順のみ（指標があるソースで質を捨てることになる）。

### D-032. ホーム吹き出しは daily_summaries テーブルで動的化（全体共通 1 本）
- **決定**: 当日 TOP5 確定後に 1 コールで一言サマリーを生成し、`daily_summaries`（date PK, summary_ja, created_at）へ upsert。フロントは当日行を表示し、無ければフォールバック文言。Slack 通知の冒頭文にも流用。
- **理由**: 追加コールは 1 日 1 回のみで枠への影響が無視できる。属性別に作り分けると属性の組み合わせ分コールが増えるため、共通 1 本とする。
- **却下案**: ハードコード継続。属性別サマリー生成（コール数増・引き算思想にも反する）。

### D-033. 新ソースの実物確認に基づく採用・調整（2026-07-04 確認）
- **決定**:
  - (a) openai/codex は `https://github.com/openai/codex/releases.atom` を採用。ただし `rust-vX.Y.Z-alpha.N` 形式の alpha ビルドが毎日大量に流れるため、**安定版のみに絞るフィルタ（`-alpha`/`-beta`/`-rc` 除外）を必須**とする（D-020 の gemini-cli nightly 対応と同型）。
  - (b) Qiita の Codex タグ slug は `codex`（記事 908 件）、Zenn は `https://zenn.dev/topics/codex/feed` で確定。
  - (c) Google News RSS 検索は「生成AI 導入事例」「生成AI ビジネス」の 2 クエリを採用（各 100 件返却・関連度良好）。クエリは **UTF-8 パーセントエンコード必須**。
  - (d) ビジネス系メディアは **ITmedia AI+**（`https://rss.itmedia.co.jp/rss/2.0/aiplus.xml`）を採用し、**PR TIMES は却下**。
- **理由**: ITmedia AI+ は AI 専門で全 20 件が対象ドメインに合致し、元記事 URL 直リンク・pubDate も正常。PR TIMES の `index.rdf` は全業種プレスリリース約 200 件の firehose で AI に絞る手段が無く、関連度フィルタ頼みになりノイズ混入リスクが大きい。
- **却下案**: PR TIMES 採用（上記の通り絞り込み不能）。codex の全リリース取り込み（alpha でフィード汚染。gemini-cli と同じ轍）。
- **補足（Google News の制約・許容済み）**: `<link>` が元記事ではなく `news.google.com/rss/articles/{不透明ID}` のリダイレクト URL のため、**他ソースとの横断重複排除（D-007 の意図）は Google News 由来記事には効かない**（同一 ID 同士では効く）。また description が薄く、enrich 入力はタイトル中心になる。導入事例・ビジネス系は他ソースと記事が被りにくいため許容する。
- （追記）**(c) の Google News 採用は D-034 により撤回**。上記の許容済み制約が、はてブ検索への置き換えで全て解消できると判明したため。(a)(b)(d) は有効なまま。

### D-034. 導入事例・ビジネス系ソースは Google News をやめ、はてブ検索 RSS を採用（2026-07-04 確認）
- **決定**: 導入事例系・ビジネス系の 2 クエリは、Google News RSS 検索ではなく**はてなブックマーク検索 RSS** で取得する。URL 形式は `https://b.hatena.ne.jp/q/{UTF-8 %エンコード済みクエリ}?mode=rss&sort=recent&target={title|text}&users=1`。初期クエリは「**AI 導入**」（`target=title`）と「**生成AI ビジネス**」（`target=text`）の 2 本（各 40 件返却・当日分あり・実測確認済み。運用しながら調整可）。
- **理由**: Google News の許容済み制約（D-033 補足）が、はてブ検索なら全て解消される。(1) `<link>` が元記事直リンクのため **D-007 の横断重複排除がそのまま機能**、(2) `hatena:bookmarkcount` により **D-031 の popularity / rank が使える**、(3) description に本文抜粋があり enrich 入力の質が上がる、(4) **既存の hatena アダプタにクエリを追加するだけ**で済み、googleNews.ts の新規実装が丸ごと不要（引き算思想にも合致）、(5) 「最低 1 ブクマ」が人間による最低限のスクリーニングとして働く。
- **却下案**: Google News 継続（網羅性は最大だが上記 4 つのデメリットを許容し続けることになる）。ハイブリッド（はてブ + Google News 併用。googleNews.ts の実装コストが残り、利点が薄い）。AIsmiley の RSS（フィードは存在するが item が 0 件で実用不可）。Ledge.ai（RSS 廃止・404）。
- **注意**: はてブ検索 `/q/` の**既定はタグ検索（`target=tag`）+ `users=3`** のため、新クエリでは `target` と `users=1` の明示指定が必須（既定のままだと「生成AI 導入事例」は 1 件しかヒットしない）。既存の「Claude Code」クエリはタグ検索のままで問題なし。トレードオフ: 1 ブクマもされない記事は拾えない（許容済み）。無関係記事の混入は残るため relevance.ts のフィルタは引き続き必須。

### D-036. 「今日の重要 TOP5」を「重要トピック」へ改称し、ボタン5グループ別の当日最重要1件ずつ・全ユーザー一律に変更（2026-07-11）
- **決定**: 固定枠の画面名を「重要トピック」に変更し、選定ロジックを刷新する。
  - (a) ホームの 5 ボタングループ（Gemini / Claude Code / Codex / バックオフィス / 経営者向け）**それぞれの当日最重要記事を 1 件ずつ**表示する（グループ別に `importance_score` 降順・`limit(1)`。最大 5 件。該当0件のグループは非表示）。
  - (b) **全ユーザー一律**。固定のグループ定義（全項目選択相当）を使い、ユーザーのチェック状態には依存しない。ホーム吹き出しの一言サマリー（`daily_summaries`・全ユーザー一律。D-032）と情報の粒度・母集団を揃える。
  - (c) グループのフィルタは設定画面のグループ定義（`SETTINGS_GROUPS`）と同じ product / audience / category。**difficulty は選定では無視**し、未 enrich（difficulty=null）記事も対象に含める。
  - (d) カード先頭の順位番号（1〜5）を廃止し、そのカードが代表する**グループ名バッジ**（例「Claude Code」）を表示する。
  - (e) バッチ（`summarize.ts` の一言サマリー・`notify.ts` の Slack 通知）も同じ選定に揃える。選定は `batch/src/lib/topTopics.ts` と `src/hooks/useArticles.ts` に実装（別パッケージのため定義は複製。唯一の情報源は `src/lib/buttonSettings.ts`）。
- **理由**: 旧方式（importance 上位 5 件をチェック中カテゴリで絞る。D-030）は、(1) 重要度スコアの採点基準が update（機能・リリース）寄りで TOP5 が機能系記事に偏る、(2) 選択が少ないと 5 件に満たない、という 2 つの不満があった。グループ別に 1 件ずつ取れば分野の多様性が構造的に保証され、静かなグループ（Codex 等）も必ず 1 枠を得る。全ユーザー一律にすることで、一言サマリーとの整合も取れ、フロントの localStorage 依存も外れる。
- **却下案**: (i) 4 カテゴリ（update/tips/business/case_study）別に 1 件ずつ（当初検討したが、ユーザーの関心軸はカテゴリよりボタングループ＝プロダクト/対象者のため不採用）。(ii) チェック中カテゴリ絞り込みの維持（上記 2 不満が残る）。(iii) 5 件に満たない分をバックフィル（多様性の意図が薄れ、チェック外カテゴリが混ざる）。
- **実装補足**: 静かなグループの記事が全体上位から漏れて多様性が損なわれるのを避けるため、単一クエリ＋クライアント集約ではなく**グループ別に並列クエリ**（各 `limit(1)`・軽量）を投げる。id/special/icon は `'top5'` を据え置き（ルーティング `/top5`・アイコン・グラデーション・空状態メッセージが依存するため、表示ラベルのみ変更）。

### D-037. product は LLM の内容判定を優先し、取得クエリ由来の値はヒント兼フォールバックに格下げ（2026-07-13）
- **決定**: enrich の LLM 出力に `product`（claude_code / gemini / codex / other）を追加し、DB 保存時は **LLM 判定を優先**する。取得クエリ由来の product はプロンプトに「収集元クエリ由来の推定値」として渡し（判断がつかない場合のみ採用する指示）、LLM 出力が欠損・不正値のときのフォールバックとしてのみ使う。再分類が発生した記事は実行ログに `product reclassified: gemini -> claude_code (URL)` 形式で出力する。
- **理由**: 従来は「どのクエリで取得したか」だけで product が確定していたが、Zenn/Qiita のトピックタグは著者任せのため誤爆する（実例: Claude Sonnet 5 の記事に著者が Gemini タグを付けており、zenn_gemini 経由で product=gemini として保存された。2026-07-12）。はてブ検索も本文に競合プロダクト名を含むだけで通過する同型の弱点を持つ。category / audience / difficulty は既に LLM 判定であり、product だけ経路決めになっている方が例外的だった。既存の 1 記事 = 1 コール（D-006）に出力フィールドを 1 つ足すだけでコール数は増えない。
- **却下案**: タイトルの競合プロダクト名検出などルールベースの再分類（今回の実例は捕捉できるが比較記事で破綻し、キーワード保守も増える）。取得クエリの絞り込み強化（著者のタグ付けを制御できない以上、根本解決にならない）。
- **副作用（許容済み）**: hatena_ai_adoption / hatena_ai_business / itmedia など従来一律 other だったソースの記事も、特定プロダクトが主題なら claude_code / gemini / codex に分類されるようになる（内容ベースとしてはむしろ正しい方向）。既存の誤分類レコードはバックフィルせず 30 日ローテーションで解消する（CLAUDE.md §4 の方針どおり）。

### D-038. 重要トピックはバッチ実行時に確定してスナップショット保存し、画面・サマリー・Slack は保存結果を読むだけにする（2026-07-16）
- **決定**: 重要トピックの選定（D-036 のグループ別最重要 1 件ずつ）は **collect バッチの最終ステップで 1 回だけ実行**し、結果を新テーブル **`daily_topics`**（date + position を PK、article_id で `articles` を FK 参照）へスナップショット保存する。画面（`useArticles`）・一言サマリー（`summarize.ts`）・Slack 通知（`notify.ts`）は選定クエリを再実行せず、**保存済みスナップショットを読むだけ**にする。画面は「最新の date のスナップショット」を表示する（バッチ前の 0〜5 時は前日分が出る）。あわせて選定クエリの同点順序を決定化する（`importance_score` 降順 nullsLast → `published_at` 降順）。
- **理由**: 従来は「直近 24 時間」のローリングウィンドウ（D-023）の**起点時刻が消費者ごとに異なっていた**（バッチ = 毎朝 5 時、画面 = ユーザーが開いた瞬間）。このため夕方以降に画面を見ると Slack が対象にした記事の大半がウィンドウ外に脱落し、(1) グループごと消える（2026-07-16 のバックオフィス欠落）、(2) Slack と画面の内容が食い違う、(3) グループ内の最重要記事（重要度 9）より低い記事（重要度 8）が表示される、という実害が出た（いずれも実データで確認済み）。また選定ロジックが `batch/src/lib/topTopics.ts` と `src/hooks/useArticles.ts` に複製されており（別パッケージのため）乖離リスクが構造的に残るうえ、同点時の順序が非決定でスナップショットなしでは同一ウィンドウでも消費者間で結果がずれうる。選定を 1 回にして結果を共有すれば、この 3 問題が同時に解消する。
- **却下案**: (i) 画面側のウィンドウ起点を「最新バッチ実行時刻」に合わせる（二重実装と同点非決定性が残り、根本解決にならない）。(ii) `daily_summaries` に選定結果の JSON 列を追加（カード表示に必要な記事カラム一式を非正規化コピーするか結局 `articles` を引き直す必要があり、正規化した別テーブルの方が素直）。(iii) `daily_topics` に記事内容を丸ごとコピー（articles と二重管理になる。FK 参照なら cleanup の 30 日削除に `on delete cascade` で自動追従でき、古いスナップショット行の掃除も不要）。
- **実装補足**: `daily_topics` は anon 読み取りの RLS を付与（`daily_summaries` と同様）。フロントは PostgREST の FK 埋め込み（`articles(...)` ）で記事カラムを 1 クエリで取得する。collect 側のスナップショット保存は fail-soft（失敗しても collect 全体を落とさない。その日は 0 行 = 空状態扱い）。ホーム画面のサブタイトル「直近 24 時間の重要記事をピックアップ。毎朝 5 時に更新」は本決定後の挙動と完全に一致する。

### D-039. 重要度スコアの採点基準をカテゴリ横断の「読者の行動・判断をどれだけ変えるか」に変更（2026-07-18）
- **決定**: enrich プロンプト（`batch/src/llm/gemini.ts`）の重要度基準を、update 語彙のアンカー（「10=破壊的変更/メジャーリリース, 7-8=重要な新機能, 4-6=通常アップデート/Tips」）から、**カテゴリを問わず「読者の行動・判断をどれだけ変えるか」で採点する 4 段階基準**に書き換える。各段階に update / tips / business / case_study それぞれのアンカー例を置き、「update 系だから高い / tips 系だから低いという判断はしない」と明示する。
- **理由**: 旧基準はスケールの目盛りがすべて update カテゴリの語彙で定義されており、7 以上に tips / 事例が入る余地が文言上なく（「4-6=通常アップデート/Tips」で tips は実質 6 が上限）、business / case_study は基準への言及ゼロだった。この update 偏重は D-036 でも重要トピックが機能系記事に偏る不満の原因として指摘済み。D-036 のグループ別選定は「グループ間」の多様性を構造で担保したが、スコア自体が偏っていると「グループ内」の代表記事選定（importance_score 降順）や重要度順ソート画面が引き続き歪む。
- **却下案**: (i) カテゴリ別の補正係数をコードで掛ける（LLM 採点の意味が崩れ、係数の根拠も持てない）。(ii) カテゴリごとに独立した採点基準を持つ（画面は全カテゴリ横断で importance_score を比較するため、スケールの互換性が必要）。
- **補足**: プロンプト変更のみでコード・スキーマへの影響なし。既存記事のスコアはバックフィルせず 30 日ローテーションで自然に入れ替わる（CLAUDE.md §4 の方針どおり）。SPEC.md §6.3 も同時更新済み。
